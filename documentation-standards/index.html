    Documentation Standards¶

    The documentation published in this portfolio follows recognized technical writing and accessibility practices.

    Writing Standards¶

    The documentation aligns with:

    • Microsoft Writing Style Guide
    • Google Developer Documentation Style Guide
    • Plain Language principles

    Documentation Principles¶

    The following principles guide every document.

    Clarity¶

    Write instructions that are easy to understand.

    Accuracy¶

    Ensure technical information is correct.

    Consistency¶

    Use consistent terminology, formatting, and structure.

    Accessibility¶

    Support screen readers and keyboard-only users.

    Maintainability¶

    Keep documentation easy to update.

    Heading Standards¶

    Each document uses:

    • One H1
    • H2 for major sections
    • H3 for subsections
    • H4 when additional hierarchy is required

    Tables¶

    Tables are used for:

    • Specifications
    • Roles
    • Responsibilities
    • KPIs
    • Configuration values

    Notes¶

    Material for MKDocs admonitions are used instead of plain note text.

    Examples include:

    • Notes
    • Tips
    • Warnings
    • Important information

    Version Control¶

    Documentation revisions are tracked using Git.

    Changes are documented through commit history and revision logs.

    Accessibility Checklist¶

    Each document is reviewed for:

    • Heading hierarchy
    • Keyboard navigation
    • Screen reader compatibility
    • Descriptive links
    • Table accessibility

    Quality Assurance¶

    Documentation is validated using:

    • Markdownlint
    • Vale
    • Manual review
    • Accessibility testing
